# enetLwip ‚Äî Portable StarterWare Project (BeagleBone + lwIP HTTP Server)

A standalone, portable import of the TI StarterWare **`enet_lwip`** example,
made buildable from a CCS workspace without depending on the StarterWare
project tree at build time. Demonstrates an lwIP-based HTTP web server on the
BeagleBone (AM335x) using the on-board CPSW Ethernet.

---

## What this project does

When flashed to a BeagleBone, the firmware:

1. Initializes the MMU and caches.
2. Brings up the CPSW Ethernet MAC (10/100 Mbps, MDIO PHY).
3. Acquires an IP address via DHCP (falls back to a static address if no
   DHCP server responds).
4. Starts the lwIP TCP/IP stack and an HTTP web server on port 80.
5. Serves a static web page baked into the firmware as a C array
   (`fsdata.c`).

On boot, the serial console prints something like:

```
StarterWare Ethernet Application. Access the embedded web page using
http://<ip address assigned>/index.html via a web browser.
...
Port 1 IP Address Assigned: 192.168.0.126
```

Browse from a PC on the same LAN to see the "lwIP ‚Äî A Lightweight TCP/IP
Stack" page.

## Portable-import notes

This project was originally a CCS "Copy Project to Workspace" of the StarterWare
example. The copy used linked resources pointing back into the StarterWare tree
and a build path full of `../../../`-relative references that only made sense
inside the StarterWare install. The following changes were applied to make the
copy self-contained while keeping StarterWare untouched:

### 1. Absolute include/library paths
In `.cproject`, every `${ORIGINAL_PROJECT_ROOT}/../../../../../../../...`
reference (compiler `--include_path`, linker `--library`, output
`--output_file`, post-build steps) was replaced with the absolute path
`C:/ti/AM335X_StarterWare_02_00_01_01/...`. The `<variableList>` block in
`.project` defining `ORIGINAL_PROJECT_ROOT` was removed.

### 2. Compiler version pinned to 20.2.7.LTS
`OPT_CODEGEN_VERSION` was set to `20.2.7.LTS` in both Debug and Release
configurations. Older StarterWare defaults (`4.9.1`, `5.0.4`) are not
installed in this CCS.

### 3. Sources copied locally
The three C sources and one config header that were linked from StarterWare
were copied into the project root:

| Source | From |
|---|---|
| `enetLwip.c` | `C:/ti/AM335X_StarterWare_02_00_01_01/examples/beaglebone/enet_lwip/enetLwip.c` |
| `lwipopts.h` | `C:/ti/AM335X_StarterWare_02_00_01_01/examples/beaglebone/enet_lwip/lwipopts.h` |
| `httpd.c` | `C:/ti/AM335X_StarterWare_02_00_01_01/third_party/lwip-1.4.0/apps/httpserver_raw/httpd.c` |
| `lwiplib.c` | `C:/ti/AM335X_StarterWare_02_00_01_01/third_party/lwip-1.4.0/ports/cpsw/lwiplib.c` |

The three `<link>` entries for `.c` files were removed from `.project`'s
`<linkedResources>` (only the `Debug/enetLwip.out` and `Release/enetLwip.out`
artifact links remain).

Headers and StarterWare libraries are still referenced by absolute path; they
are not copied.

### 4. Debug/Release caches cleared
The generated `Debug/` and `Release/` directories were deleted so that CCS
regenerates `makefile`, `subdir_rules.mk`, etc. from the new `.cproject`.

## Building in CCS

1. Launch CCS 12.8.1.
2. Select the workspace containing this directory
   (`C:/D/MY/DEV/TI-CCS-IDE/Workspace_12`).
3. The `enetLwip` project should appear in **Project Explorer**. If not,
   **File ‚Üí Import ‚Üí CCS Projects** and point at this folder.
4. Right-click `enetLwip` ‚Üí **Build Project** (or `Ctrl+B`).

Expected output ends with:

```
**** Build Finished ****
```

The binary lands at
`C:/ti/AM335X_StarterWare_02_00_01_01/binary/armv7a/cgt_ccs/am335x/beaglebone/enet_lwip/Debug/enetLwip.bin`.

### If the build fails after editing `.cproject`

Close the project, delete the `Debug/` and `Release/` folders, then
re-open the project. This forces CCS to regenerate the makefiles from the
new configuration.

## Running on the BeagleBone

1. Connect the BeagleBone via USB (serial debug) and Ethernet.
2. Identify the COM port assigned to the USB serial (e.g. `COMx`).
3. In CCS, use the pre-configured launch (if present) or create a debug
   configuration targeting the BeagleBone's JTAG emulator.
4. Load and run the `.out` (or flash the `.bin` via the StarterWare
   flashing scripts).
5. Watch the serial console ‚Äî note the assigned IP address.
6. From a PC on the same LAN, browse to `http://<that-ip>/index.html`.

### Console menu

After the web server starts, the console exposes a small PHY
reconfiguration menu (`Configure Phy of a Port`). This only changes the
link advertisement to the Ethernet switch ‚Äî it does not change application
behavior. The default selections (autoneg, advertise all speeds) give the
fastest stable link.

## Troubleshooting

| Symptom | Likely cause |
|---|---|
| `cannot open source file "xxx.h"` after editing `.cproject` | Stale `Debug/subdir_rules.mk`; delete `Debug/` and `Release/` and rebuild |
| Linker cannot find `drivers.lib` / `system.lib` / etc. | StarterWare support libraries were never built; build them first |
| DHCP trial fails and IP is `0.0.0.0` | No DHCP server reachable; the static fallback should kick in after retries |
| `Phy does not support Gigabyte...` | Expected ‚Äî BeagleBone's onboard PHY is 10/100 only |
| Browse to IP shows nothing | Firewall on PC blocking port 80, or wrong subnet ‚Äî verify `ipconfig` and ping the board first |
